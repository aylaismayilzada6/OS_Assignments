System Overview 
The system goal is to  manage three replicas of a data resource (NUM_REPLICAS 3) using a Writer-Preference synchronization strategy. The core logic ensures that while multiple readers can access different replicas to balance the load, any pending writer request blocks new readers from starting, preventing writer starvation. 


The implementation utilizes POSIX semaphores and shared state flags to coordinate thread access. The primary goal is to prevent data corruption and ensure that writers are not starved by a constant stream of readers.


To satisfy the requirement of distributing readers evenly, the system implements the function chooseReplica() that iterates through the readCount array and returns the index of the replica with the fewest active readers.


The solution utilizes two semaphores and several shared state variables to maintain thread safety:
* sem_t mutex: A binary semaphore (mutex) used to protect shared metadata, including readCount, totalReaders, and writerWaiting flags.
* sem_t writeLock: A semaphore that controls access to the actual data replicas. It is acquired by the first reader to block writers and by the writer to block all readers.
* writerWaiting: A state flag that, when set to 1, forces arriving readers into a retry loop, effectively giving the writer priority.
Reader Threads
When a writer intends to access the files, it sets a writerWaiting flag. Subsequent readers check this flag and must wait until the writer completes its task before they can proceed.
* Before reading, the thread enters a while(1) loop to check if writerWaiting is active. If a writer is waiting, the reader releases the mutex and sleeps briefly before retrying.
* Once cleared to proceed, it calls chooseReplica() to identify the best target.
* If it is the totalReaders == 1 (the first reader), it acquires writeLock to prevent writers from entering.
* After a simulated read (sleep(1)), it decrements the counters. If it is the last reader (totalReaders == 0), it releases writeLock.
Writer Thread
To maintain uniformity, the writer performs an "atomic-style" update by iterating through all replicas and applying the same newContent buffer. Readers are blocked during this phase to ensure they never read partially updated or stale data.
*  It immediately sets writerWaiting = 1 to prevent new readers from joining the queue.
* It calls sem_wait(&writeLock), which blocks until all current active readers have finished.
* It updates all three replicas simultaneously using a for loop, ensuring no reader sees partial or mismatched data.
* After updating, it resets writerWaiting to 0 and releases writeLock to allow readers to resume.
Logging 
The system maintains a log.txt file to provide a real-time audit update of every operation. The system's logging mechanism records the action type, thread and replica IDs, load distribution across files, and the current writer status alongside replica content to ensure full traceability of every synchronization event.
